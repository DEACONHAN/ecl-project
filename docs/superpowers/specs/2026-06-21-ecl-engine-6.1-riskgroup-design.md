## 1. 目标

在 `ecl-engine` 模块中实现风险分组引擎（6.1），负责根据借据的 6 维属性将每笔借据匹配到对应的风险分组。

后续 7 个引擎（6.2~6.8）在此基础上扩展，共享同一套核心接口和上下文对象。

## 3. 核心接口和数据结构

### 3.1 EclEngine 接口

```java
package com.bank.ecl.engine.core;

public interface EclEngine {
    EngineType getType();
    void execute(JobContext ctx);
}
```

所有 8 个引擎实现此接口。`getType()` 返回引擎类型枚举，`execute()` 接收任务上下文并修改其中的借据中间结果。

### 3.2 JobContext

任务级上下文，贯穿一次 ECL 计算任务的全部 8 个引擎。

| 字段 | 类型 | 说明 |
|------|------|------|
| jobId | String | 任务 ID |
| schemeId | String | 绑定的 ECL 方案 ID |
| trialMode | boolean | true=试算，false=正式跑批 |
| defaultCcf | double | 方案 CCF 缺省值 |
| defaultLgd | double | 方案 LGD 缺省值 |
| discountRate | double | 折现率 |
| customers | List\<CustomerContext\> | 按客户分批的上下文列表 |

### 3.3 CustomerContext

客户级上下文，一个客户下的所有借据、额度、押品信息。

| 字段 | 类型 | 说明 |
|------|------|------|
| customerId | String | 客户 ID |
| assets | List\<AssetInput\> | 该客户下所有借据 |

预留字段（后续引擎使用）：`facilities`（额度信息）、`collaterals`（押品信息）。

### 3.4 AssetInput

借据输入及引擎逐层填充的中间结果。

| 字段 | 类型 | 来源 | 说明 |
|------|------|------|------|
| assetId | String | 入参 | 借据 ID |
| customerId | String | 入参 | 所属客户 ID |
| businessLine | String | 入参 | 业务条线 |
| customerType | String | 入参 | 客户类型 |
| productType | String | 入参 | 产品类型 |
| industryCode | String | 入参 | 行业代码 |
| regionCode | String | 入参 | 地区代码 |
| collateralType | String | 入参 | 担保类型 |
| groupId | String | 6.1 输出 | 匹配到的分组 ID |
| groupName | String | 6.1 输出 | 分组名称 |
| groupException | String | 6.1 输出 | "Y"=异常（命中兜底） |

预留字段（后续引擎使用）：`lastStage`、`stageResult`、`pdDetails`、`totalEad`、`lgdValue`、`eclValue` 等。

## 4. RiskGroupEngine 详细设计

### 4.1 类定义

```java
package com.bank.ecl.engine.riskgroup;

@Component
public class RiskGroupEngine implements EclEngine {

    private final RiskGroupDetailMapper detailMapper;
    private final RiskGroupMapper groupMapper;

    @Override
    public EngineType getType() {
        return EngineType.RISK_GROUP;
    }
}
```

- 通过构造注入获取 `RiskGroupDetailMapper` 和 `RiskGroupMapper`（来自 `ecl-data` 模块）
- `@Component` 注解使其被 Spring 自动扫描

### 4.2 execute() 流程

```
1. 从 Mapper 一次性加载 scheme 下：
   a. 所有分组明细规则 → List<RiskGroupDetailEntity>（按 priority ASC 排序）
   b. 所有分组主数据 → Map<groupId, RiskGroupEntity>

2. 对每个 CustomerContext → 每个 AssetInput：
   a. 调用 matchGroup() 匹配规则
   b. 若命中 → 设置 groupId
   c. 若全未命中 → 设置 groupId = "GRP_DEFAULT"，groupException = "Y"
   d. 从 groupMap 取 groupName 设置到 asset

3. 引擎不抛异常 — 匹配失败以兜底分组 + exception flag 处理，
   保证后续引擎链不会因个别借据中断
```

### 4.3 匹配算法

**6 维 AND 匹配，NULL 或空字符串 = 通配：**

```java
private String matchGroup(AssetInput asset, List<RiskGroupDetailEntity> rules) {
    for (RiskGroupDetailEntity rule : rules) {
        if (matchDimension(asset.getBusinessLine(), rule.getBusinessLine())
            && matchDimension(asset.getCustomerType(), rule.getCustomerType())
            && matchDimension(asset.getProductType(), rule.getProductType())
            && matchDimension(asset.getIndustryCode(), rule.getIndustryCode())
            && matchDimension(asset.getRegionCode(), rule.getRegionCode())
            && matchDimension(asset.getCollateralType(), rule.getCollateralType())) {
            return rule.getGroupId();
        }
    }
    return "GRP_DEFAULT";
}

private boolean matchDimension(String assetValue, String ruleValue) {
    if (ruleValue == null || ruleValue.isBlank()) return true;
    if (assetValue == null || assetValue.isBlank()) return true;
    return ruleValue.equals(assetValue);
}
```

**关键行为：**
- 规则按 `priority ASC` 排序，命中第一条即停止
- ruleValue 为 NULL/空 → 该维度通配，不限制
- assetValue 为 NULL/空 → 该维度通配，不限制
- 全部规则未命中 → 兜底分组 `GRP_DEFAULT`

### 4.4 数据加载

```java
private List<RiskGroupDetailEntity> loadRules(String schemeId) {
    return detailMapper.selectList(
        new LambdaQueryWrapper<RiskGroupDetailEntity>()
            .eq(RiskGroupDetailEntity::getSchemeId, schemeId)
            .orderByAsc(RiskGroupDetailEntity::getPriority));
}

private Map<String, RiskGroupEntity> loadGroupMap(String schemeId) {
    List<RiskGroupEntity> groups = groupMapper.selectList(
        new LambdaQueryWrapper<RiskGroupEntity>()
            .eq(RiskGroupEntity::getSchemeId, schemeId));
    return groups.stream().collect(Collectors.toMap(
        RiskGroupEntity::getGroupId, Function.identity()));
}
```

**加载策略：** 规则一次性全量加载到内存，后续逐借据在内存中匹配，不逐笔查数据库。

## 5. 依赖关系

```
ecl-engine
  ├── ecl-parameter（传递依赖）
  │   └── ecl-data（传递依赖）
  │       └── ecl-common（传递依赖）
  └── Spring Context（@Component 自动装配）
```

引擎层直接使用 Mapper 查询，不经过 Service 层，因为引擎只需要全量加载 + 内存匹配，不涉及 CRUD 业务逻辑。

## 6. 错误处理策略

| 场景 | 处理 |
|------|------|
| schemeId 下无分组规则 | 所有借据→GRP_DEFAULT，groupException="Y" |
| schemeId 下无分组主数据 | groupName 设为 null |
| 个别借据 6 维全空 | 正常匹配逻辑，最终命中 GRP_DEFAULT |
| 数据库异常 | 由 MyBatis-Plus 自然抛出，上层统一处理 |

## 7. 单元测试

### 7.1 测试范围

- Mapper 层使用 Mock，不连接真实数据库
- 覆盖：正常分组匹配、部分维度匹配、通配匹配、兜底分组、全量规则无命中

### 7.2 测试场景

| 场景 | 预期 |
|------|------|
| 规则列表为空 | 所有借据→GRP_DEFAULT，exception=Y |
| 单条规则精确命中 | groupId 匹配，exception=null |
| 部分维度 NULL（通配） | 匹配成功 |
| 多条规则命中第一条（priority） | 返回 priority 最小的那条 |
| 全部规则无命中 | GRP_DEFAULT，exception=Y |
| 分组主数据存在 | groupName 正确填充 |

## 8. 暂不实现

以下内容属于后续引擎（6.2~6.8）范围，本次不实现：

- `EngineDispatcher` 调度器
- `JobContext` 中除 schemeId 外的参数动态加载（defaultCcf/defaultLgd/discountRate 由方案表自行查取）
- `CustomerContext` 中的 facilities/collaterals 字段
- `AssetInput` 中非 6.1 输出的预留字段
- `ConditionEvaluator` 的 JSON 条件匹配（6.1 使用简单的 6 维 field 匹配，无需 JSON 条件解析）
