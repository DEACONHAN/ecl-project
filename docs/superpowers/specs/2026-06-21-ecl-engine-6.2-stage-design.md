## 1. 目标

在 `ecl-engine` 模块中实现阶段划分引擎（6.2），根据借据的逾期天数、评级、五级分类、违约标识、舆情等信息，通过 FORWARD（向前判定）+ ROLLBACK（回跳校验）两步法判定该借据所处的 IFRS 9 三阶段。

## 6. StageConditionEvaluator

### 6.1 职责

解析 `tbl_stage_rule.conditions` JSON 字符串，评估借据是否满足条件。专用于阶段划分引擎的 JSON 条件匹配，不修改 `ecl-common` 的 `ConditionEvaluator`。

### 6.2 支持的 JSON 条件类型

| 条件类型 | JSON 结构 | 说明 |
|---------|----------|------|
| 范围匹配 | `{"overdue_days": {"min": 31, "max": 90}}` | 逾期 31-90 天 |
| 枚举匹配 | `{"five_category": {"in": ["次级","可疑","损失"]}}` | 五级分类命中 |
| OR 逻辑 | `{"or": {"sub_cond_1": ..., "sub_cond_2": ...}}` | 任一子条件满足即通过 |
| AND 逻辑 | `{"and": {"sub_cond_1": ..., "sub_cond_2": ...}}` | 所有子条件满足才通过 |
| CRR 下降 | `{"crr_drop": {"system": "CRR", "use_lookup": true}}` | 查 crrDropMap 判断阈值 |
| 等值匹配 | `{"default_flag": true}` 或 `{"repayment_status": "normal"}` | 精确匹配 |
| 兜底 | `{"default": true}` | 无条件通过 |

### 6.3 核心方法

```java
public class StageConditionEvaluator {

    /**
     * @param conditionsJson 阶段规则的 conditions JSON 字符串
     * @param asset          借据（提供字段值）
     * @param crrDropMap     当前评级 → 下降阈值（可为 null）
     * @return true=条件满足
     */
    public static boolean evaluate(String conditionsJson,
                                   AssetInput asset,
                                   Map<String, Integer> crrDropMap);
}
```

### 6.4 匹配逻辑

```
解析 JSON → 遍历顶层 key：
  ├─ "or"       → 任一子条件 evaluateSub() 通过 → true（全部失败 → false）
  ├─ "and"      → 所有子条件 evaluateSub() 通过 → true（任一失败 → false）
  ├─ "crr_drop" → 查 crrDropMap[asset.crrRating] >= asset.ratingDropLevels
  ├─ "default"  → 无条件 true
  └─ 其他 key    → 根据 value 类型分发：
       ├─ JSONObject {"min":..., "max":...} → 范围匹配
       ├─ JSONObject {"in":[...]}           → 枚举匹配
       └─ 标量值                             → 等值匹配
           └─ null 或空字符串                → 无条件 true（通配）
```

## 7. StageEngine

### 7.1 类定义

```java
@Component
@RequiredArgsConstructor
public class StageEngine implements EclEngine {

    private final StageRuleMapper stageRuleMapper;
    private final CrrRatingDropRuleMapper crrDropRuleMapper;

    @Override
    public EngineType getType() {
        return EngineType.STAGE;
    }
}
```

- 构造注入 `StageRuleMapper` 和 `CrrRatingDropRuleMapper`
- `@Component` 注解使其被 Spring 自动扫描

### 7.2 execute() 流程

```
1. 从 schemeId 一次性加载：
   a. FORWARD 规则 → List<StageRuleEntity>（priority ASC 排序）
   b. ROLLBACK 规则 → List<StageRuleEntity>
   c. CRR 下降阈值 → Map<currentRating, dropThreshold>（按 groupId 分组，实际使用时不区分 group）

2. 对每个 CustomerContext → 每个 AssetInput：
   a. 取该借据的 groupId → 过滤出对应分组的 CRR 阈值
   b. 调用 determineStage(asset, forwardRules.filter(groupId), rollbackRules.filter(groupId), crrDropMap)
   c. asset.setStageResult(result)
```

> **注意**：FORWARD 和 ROLLBACK 规则都按 group_id 过滤——只有匹配借据 groupId 的规则才参与判定。

### 7.3 determineStage() 两步法

```
Step 1 — FORWARD 向前判定：

  遍历 forwardRules（已按 priority ASC）：
    条件满足 → targetStage = rule.stageTo, exceptionFlag = false, 记录 triggerType, break
    全部未命中 → targetStage = STAGE_1, exceptionFlag = true

Step 2 — ROLLBACK 回跳校验（仅阶段改善时触发）：

  if targetStage.ordinal() < lastStage.ordinal():
    查 rollbackRules → stageFrom=lastStage, stageTo=targetStage
      ├─ 命中且条件满足 → 允许回跳，返回 targetStage
      └─ 未命中或条件不满足 → 禁止回跳：
           返回 lastStage, exceptionFlag = true, triggerType = "ROLLBACK_BLOCKED"
  else:
    直接返回 targetStage
```

### 7.4 关键约束

- **禁止 Stage 3 → Stage 1 直达**：FORWARD 判定时 Stage 3 规则优先级最高；如需回跳必须经 Stage 2 中转（ROLBACK 规则 enforce）
- **FORWARD 规则顺序**：按 priority ASC，命中第一条即停止
- **CRR 下降**：上游预计算 ratingDropLevels，引擎只需查表对比阈值
- **引擎不阻断**：匹配失败以兜底（STAGE_1）+ exception flag 处理

## 8. 错误处理

| 场景 | 处理 |
|------|------|
| schemeId 下无 FORWARD 规则 | 所有借据→STAGE_1，exceptionFlag=true |
| schemeId 下无 ROLLBACK 规则 | 阶段改善时视为校验失败，保持原阶段 |
| conditions JSON 解析失败 | 记录日志，视为不满足条件 |
| 借据无 lastStage（首次计算） | 视为 STAGE_1，不需要回跳校验 |
| 借据无 groupId（6.1 兜底） | 使用 GRP_DEFAULT 对应的规则（如有） |

## 9. 测试场景

| 场景 | 预期 |
|------|------|
| FORWARD 规则列表为空 | 所有借据→STAGE_1，exceptionFlag=true |
| 单条 FORWARD 规则命中 | stageTo 匹配，exceptionFlag=false |
| 多条 FORWARD 规则命中第一条（priority） | 返回 priority 最小的那条的 stageTo |
| 全部 FORWARD 未命中 | STAGE_1 兜底，exceptionFlag=true |
| 阶段改善 + ROLLBACK 校验通过 | 返回改善后的 targetStage |
| 阶段改善 + ROLLBACK 校验失败 | 保持原阶段，exceptionFlag=true |
| 阶段不变或恶化 | 跳过 ROLLBACK 校验，直接返回 targetStage |
| 无 lastStage（首次计算） | 视为 STAGE_1，FORWARD 判定决定结果 |
| OR 条件 | 任一子条件满足即通过 |
| CRR 下降条件 | 下降级数 >= 阈值 → 满足 |
| "default": true 兜底条件 | 无条件通过 |

## 10. 暂不实现

- `StageEngine` 不修改 `ecl-common` 的 `ConditionEvaluator`（在 engine 模块内新建 `StageConditionEvaluator`）
- 不实现 `observationDays` 观察期的自动计算（ROLBACK 规则中 `observation_days` 字段预留，当前仅在 conditions 中使用 `holdback_days_met` 等条件间接判断）
